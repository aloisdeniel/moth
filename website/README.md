# moth website

Public website (landing page + docs) for moth — milestone 09. A fully static
Astro site with Starlight for the `/docs` section. No trackers, no external
requests: fonts, search, and all assets are self-hosted.

## Commands

Run inside `website/`:

- `npm ci` — install dependencies
- `npm run dev` — dev server with hot reload
- `npm run build` — static build into `dist/`
- `npm run preview` — serve the built `dist/` locally
- `npm run check-links` — internal link/asset integrity over `dist/` (after a build)
- `npm run lighthouse` — Lighthouse budget (`lighthouserc.json`) over `dist/`
- `npm run screenshots` — regenerate the landing screenshots (see below)

From the repo root, the same wrapped as make targets:

- `make website` — `npm ci && npm run build` in `website/`
- `make website-check` — internal link check over the built site
- `make website-screenshots` — build `bin/moth` and regenerate screenshots

## Deploy & GitHub Pages configuration

Deploy is automated: `.github/workflows/pages.yml` builds `website/` and
publishes it to GitHub Pages on every push to `main` that touches `website/`
(and on manual `workflow_dispatch`). Quality gates run separately in
`.github/workflows/website.yml`.

The base path and origin are driven by **repository variables** (Settings →
Secrets and variables → Actions → Variables) so no domain is hard-coded:

| Variable | Default | Meaning |
|---|---|---|
| `WEBSITE_SITE` | `https://aloisdeniel.github.io` | absolute origin the site is served from |
| `WEBSITE_BASE` | `/moth` | path the site is mounted at |
| `WEBSITE_DOMAIN` | _(unset)_ | custom domain; when set, the deploy writes `dist/CNAME` |

The defaults deploy to **project pages** (`https://aloisdeniel.github.io/moth`),
which works today with no domain registered. To move to a **custom domain**:

1. register the domain and point its DNS at GitHub Pages,
2. set `WEBSITE_DOMAIN=<domain>`, `WEBSITE_SITE=https://<domain>`, and
   `WEBSITE_BASE=/` in the repository variables.

The workflow then writes `dist/CNAME` automatically — no placeholder domain is
ever committed to the repo. Locally you can reproduce either layout:

```sh
# project pages (current default)
WEBSITE_SITE=https://aloisdeniel.github.io WEBSITE_BASE=/moth npm run build
# custom domain (root)
WEBSITE_SITE=https://<domain> WEBSITE_BASE=/ npm run build
```

All internal links go through Astro's `BASE_URL`, so both layouts work.

Versioned deploys on release tags are a documented TODO in `pages.yml`
(plan/09: versioned docs snapshots start at v1.0).

## Screenshots

The landing page references seeded product screenshots in
`public/screenshots/`, generated by `scripts/screenshots.mjs` from a real,
throwaway moth instance:

```sh
make website-screenshots         # from the repo root (builds bin/moth first)
# or, with bin/moth already built and deps installed:
cd website && npm run screenshots
```

It creates three demo projects with distinct branding, seeds deterministic
analytics (`moth admin seed-analytics --seed 1`), and captures the projects
list, the themed login (light + dark), and the analytics dashboard through
the real admin SPA — reusing the Playwright setup from `web/admin`. It is
deterministic (fixed seed, viewport, reduced motion) and re-runnable; the
outputs are committed so the site builds without regenerating them.

Requirements: `bin/moth` (`make build`), Playwright chromium in `web/admin`
(`npx playwright install chromium`), and `sharp` in `website` (`npm ci`).
When the binary or browser is unavailable the script degrades to clearly
**labeled placeholder** PNGs at the right dimensions so the build still
resolves; set `MOTH_SCREENSHOTS_STRICT=1` (as CI does) to fail instead.

## Quality gates (CI)

`.github/workflows/website.yml`:

- **Internal links** — `scripts/check-links.mjs` over `dist/` (every
  href/src/srcset must resolve, base-path aware).
- **External links** — `lychee` over the built HTML.
- **Lighthouse** — `lighthouserc.json` asserts performance / accessibility /
  SEO ≥ 0.95 on the landing page and a docs page (best-practices ≥ 0.9 as a
  warning). Run locally with `npm run lighthouse` (needs Chrome).
- **Screenshots** — the pipeline above runs under `MOTH_SCREENSHOTS_STRICT=1`
  and uploads the images as an artifact.

Dart docs-snippet compilation (`dart analyze` over fenced `dart` blocks
against the SDK) is a planned gate; the docs are still placeholder content,
so it is not wired yet.

## Design

`/DESIGN.md` at the repo root is the visual contract. It is materialized as
`src/styles/tokens.css` (colors both schemes, type scale, spacing, radii,
motion), imported by the landing page layout and by Starlight through
`customCss` (`src/styles/starlight.css` maps the tokens onto Starlight's
CSS custom properties). Deviations are made by amending `DESIGN.md` first.

### Fonts

| Face | Weights | Source | License | Hosting |
|---|---|---|---|---|
| Satoshi | 400 / 500 / 700 | Fontshare (Indian Type Foundry) | Fontshare Free Font License (ITF) | Fontshare CDN |
| Cascadia Code | 400 / 700 | github.com/microsoft/cascadia-code v2407.24 | SIL OFL 1.1 | self-hosted, `src/fonts/` |

Cascadia Code is bundled as woff2 under `src/fonts/`, referenced with
relative `url()`s so Vite fingerprints them and any `base` works. **Satoshi
is loaded from Fontshare's CDN** (`api.fontshare.com`), not bundled: the ITF
license permits CDN hosting but not redistributing the font files. The
`<link>` is non-render-blocking (`media="print"` + `onload` swap), so if the
CDN is unreachable `--moth-font-text` falls back to the system UI font.
`font-display: swap` everywhere. Beyond that font, the site makes no external
requests — no analytics, no trackers.

## Content layout

- `src/pages/index.astro` — landing page (outside Starlight), shell +
  content stubs; hero copy per `plan/09-website.md`.
- `src/content/docs/docs/` — Starlight content, mounted at `/docs/…`
  (Starlight routes mirror the directory, hence the nested `docs/`).
- Generated references (CLI, proto API) are copied into the content tree by
  CI in a later stage — never hand-edited here.
