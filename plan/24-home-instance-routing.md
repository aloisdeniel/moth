# Milestone 24 — Home-Instance Routing

## Goal

Give every user a **home instance**: the instance that first creates the account
owns it forever — its user row, password hash, identities, refresh tokens, email
tokens, subscriptions, push devices, events. All of a user's traffic lands there.
The fabric's job (23) was to make every instance *able* to serve every project;
this milestone decides **which instance serves which user** and makes wrong-door
requests self-correct: a login arriving at the wrong instance is redirected to the
home instance in one round trip, and every minted JWT carries the home instance so
clients never guess twice.

No data moves. Routing is a lookup, not a migration.

## Model

- `user_locator` (main) — the only cross-instance index in the system, and it is a
  **locator, not a mirror**: `(project_id, identity_kind, identity_hash)` →
  `instance_id, user_id`. `identity_kind` covers `email`, `google`, `apple` (the
  milestone-04 provider subjects), later `stripe`/store transaction ids for webhook
  routing. Identifiers are stored **hashed** (SHA-256, like tokens) — the main can
  answer "where does this credential live" without holding a readable user list.
- `locator_outbox` (regional) — pending locator writes (signup, identity link,
  email change, account deletion) queued when the main is unreachable, flushed in
  order on reconnect. Locator updates are idempotent upserts keyed by identity.
- JWT: the claim set (`internal/jwt.Claims`) gains `mih` (moth instance home) — the
  home instance's `base_url` (and instance id in `mih_id`). Standalone instances
  omit it, so existing tokens and verifiers are untouched.

## Deliverables

### Homing on first contact

- **Signup homes the user at the receiving instance.** Whichever instance the
  client chose (nearest, per milestone 25) executes the milestone-02 signup
  locally, then registers the identity in the main's locator (through the outbox).
  The home assignment is permanent; there is no rebalancing.
- **Social first-sign-in** (04) homes identically — the instance that completes
  the provider exchange creates the user locally and registers the `google`/
  `apple` subject in the locator.
- Locator conflicts (the same email signing up on two instances during a main
  outage) resolve **first-write-wins** at the main; the loser's signup is rolled
  back on outbox flush with a reason-coded cleanup, and the affected session is
  revoked — rare, bounded, and auditable rather than silently merged.

### Wrong-door correction

- Every `moth.auth.v1` credential RPC (login, password reset request, email
  verification resend…) hitting an instance that has no local row asks the main's
  locator once, then answers with the existing ErrorInfo mechanism: reason
  `WRONG_INSTANCE`, metadata `home_url`. The SDK (25) retries against `home_url`
  transparently; grpcurl users see an honest, actionable error.
- The receiving instance **redirects, never proxies**, credential traffic:
  proxying would put plaintext passwords through a middleman instance and hide
  real latency from the client. One extra round trip on the first contact from a
  new device is the entire cost.
- Locator misses (main unreachable) degrade explicitly: the instance serves its
  local users normally and answers unknown identities with `UNAVAILABLE` +
  `LOCATOR_UNREACHABLE` rather than a false "no such account".

### Home-stamped tokens & local sessions

- Access tokens minted by an instance carry `mih`; refresh tokens, sessions, and
  rotation state are purely local rows on the home instance — refresh at the wrong
  instance is simply `WRONG_INSTANCE` (the token hash cannot match anywhere else).
- Hosted pages and emails (verify, reset, confirm-email) are generated by the home
  instance and link to the home instance's `base_url` — email tokens are local, so
  the links must be too. The per-instance `base_url` from 23 replaces the single
  configured public URL in link building.

### Webhook routing

- Store notifications (App Store / Play / Stripe, milestones 11/17) keep their
  single configured URL — the main's. The main routes each notification via the
  locator (transaction/customer ids are registered as locator identities when a
  subscription is created) and forwards it over `moth.cluster.v1` to the home
  instance, which processes it with the existing idempotent pipeline. Unroutable
  notifications are stored and retried (they may precede the outbox flush that
  registers their id).

### Developer-backend surface

- `moth.server.v1` (secret key) is served by every instance against **local**
  users in this milestone; `Introspect` works anywhere (JWKS is global). Documented
  interim: point your backend at the main for introspection and at the home
  instance (from `mih`) for user-specific calls; milestone 26 adds main-side
  fan-out so one endpoint answers for the whole cluster.

## Key design points

- **Sticky by birth, forever.** A home instance is an immutable property of the
  account. This is what keeps "no replication" honest: every user-owned row has
  exactly one possible location, so there is no synchronization problem to solve —
  only a lookup to serve. Migration tooling, if ever wanted, is an explicit future
  admin action, not a runtime behavior.
- **The locator is minimal on purpose.** Hashed identity → instance id is the
  entire schema. It cannot answer "list users", cannot leak emails, and rebuilding
  it is a replay of each instance's identities table — it is an index, and
  disposable like one.
- **The token is the routing table for clients.** Embedding `mih` means a client
  holding a session needs no directory, no probe, and no main to find its data —
  the same offline-verifiable self-sufficiency the ES256 JWT already gives
  developer backends.
- **Wrong-door is a first-class flow, not an error path.** Devices travel,
  directories go stale, apps restore from backup on new hardware. `WRONG_INSTANCE`
  + retry is designed to be hit routinely and cost one round trip, which is why it
  rides ErrorInfo like every other machine-readable auth outcome (02).

## Acceptance criteria

- Signup on instance B: user row, refresh token, and events exist on B only; the
  main's locator maps the hashed email to B; the JWT carries B's `mih`.
- Login for that email on instance A returns `WRONG_INSTANCE` with B's URL in one
  locator round trip; login on B succeeds; refresh presented to A is
  `WRONG_INSTANCE` without a locator call (local-miss on token hash).
- Password reset requested via A for a B-homed user: redirect, email sent by B,
  link targets B's `base_url`, reset completes on B.
- Kill the main: existing users on both instances log in, refresh, and reset
  passwords normally; a brand-new signup on B succeeds and its locator write
  flushes when the main returns; unknown-identity logins fail with
  `LOCATOR_UNREACHABLE`, not "not found".
- Same email signs up on A and B while the main is down: on flush, exactly one
  account survives, the other is rolled back and its session revoked; the audit
  log (10) records the conflict.
- A store notification for a B-homed subscription posted to the main is processed
  on B exactly once (idempotency asserted on double-delivery).
- Standalone instance: no `mih` claim, no locator calls, all tests from 02/04
  pass unchanged.

## Out of scope

Client-side nearest-instance selection and transparent retry UX (25), admin
cross-instance views and secret-key fan-out (26), account migration between
instances, geo-IP databases (the client picks its instance; the server never
guesses location), and merging conflicting accounts (first-write-wins rollback is
the policy).
